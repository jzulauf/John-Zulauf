Glibc Memcpy Optimiziation for Geode LX

Overview

Introduction

This document will cover glibc memcpy optimization investigations for the Geode LX (Castle) / 5536 platform. Algorithms tuned for Geode LX will be tested against the i486 and i586 memcpy implementations. The methodology and results will be presented

Optimization and Power

The effort to optimize for the Geode LX is derived from higher level group objectives:

  • Boot2fun:

reducing the time for our customers from receiving a development kit for LX  to creating targeted and optimized solutions for their platforms and customers (i.e. the time from when they first boot a Geode to when it’s doing something cool, interesting or “fun.”

 

  • Power Management:

This is a sub-goal of boot2fun but is specifically targeted in the goals of our software optimization.

 

  • Expanded Geode LX opportunities

By optimizing LX performance, LX increase the value proposition both in absolute performance (vs. higher cost/power solutions) or performance/$.

 

The power impact specific optimizations falls into two high level categories, direct and indirect power savings.

 

  • Direct Power Savings

The optimization of the algorithm results in fewer instructions executed and lower dynamic power (i.e. CMOS state change power).  This sort of optimization is typically related to a specific, complex, and application-specific algorithm.

 

  • Indirect Power Savings

Here power is saved not by running fewer instructions but by improving the effective CPI of the processor, allowing for more time in deeper power managed states, or allowing for a lower-clocking/lower-voltage parts use.

 

The memcpy routine falls into this latter category. The standard POSIX memcpy copies a string of bytes from one location in memory to another, with the assumption that the regions do not overlap (Overlapping regions are handled by memove.) In Linux this is implemented by glibc with a variety of compile time and runtime implementations linked or inserted by the compiler (the complexity of this is beyond the scope of this documents)

Goals

These goals for this study are:

  • Identify the potential for performance increases by optimization in a specific narrow context – memcpy performance in isolation glibc implementation vs. LX targeted
  • Measure the performance impact on “out of cache” copies.
  • Test both “in situ” (on the platform) and “en vitro” (in the simulation) to obtain the highest optimization
  • Identify the complexity of insertion of the LX optimizations into glib

Algorithmic Approach

The algorithms were based on the initial code fragement from Steve Kommrusch:

 

        mov esi, offset data0_read
        mov edi, offset data0_write
        ; Copy 80,000 bytes of memory using rep movsd loops
        mov ebx, 625
        mov eax,esi
        add eax, 192
        mov ecx, 32
     loop1:
        ; Prefetch lines we'll need a couple loops later
        lea eax,[eax+32]
        PREFETCHT0_EAX
        lea eax,[eax+32]
        PREFETCHT0_EAX
        mov cl, 32
        lea eax,[eax+32]
        PREFETCHT0_EAX
        ;; Move 128 bytes
        rep movsd
        lea eax,[eax+32]
        PREFETCHT0_EAX
        dec ebx
        jnz loop1

 

The core of this approach is unrolling the single repmov into several which span integer cachelines, which are prefetched prior to the repmovs.  RedCapFX (Geode GX optimization “tiger” team) identified 2,3, and 4 cacheline unrolling as the optimal.

Additional optimization was do to minimize the call stack overhead (stack frame, local variable stack space, and register preservation), and the overhead for the  “number of bytes less than N cachelines” case.

 

Note that no effort is made to avoid overfetching, as prefetch can cause neither page faults, nor page permission violations.

 

Three algorithmic variations survived this weeding, all of which eschew the use of the “LEA” instruction above.

  • Lx64 – lx_memcpy_noleal.s
    • Two cacheline unroll with a two cacheline prefetch
  • Lx64si0 lx_memcpy_si0.s
    • This is a Lx64 derivative based on inputs from Steve Thompson in Silicon Engineering reviewing the code and the Verilog
    • Notably this prefetches only 1 cacheline ahead and doesn’t prefetch the initial source address.  This reflects the internal limit of the LX of 3 outstanding prefetch transactions.
  • Lx128 – lx_memcpy_noleal_128.s
    • A four cachline unroll of Lx64 above.

 

 Build Process

All programs are created with “make”.  The default make target “all” creates both utest and ptest (see below).  The clean “target” restores the directory to the pristine state.  Note that no include file rules are included in the Makefile.

Unit Testing

Each algorithm was tested using utest (utest.c) for correctness in implementing the memcpy function. This unit testing was critical in the development of the replacement algorithms as it unveiled defects impact performance both positively and negatively.  At each refinement of the optimized routines utest was re-run to verify correctness.

 

When successful utest exits with a return code of 0 (POSIX stand for success). If errors are discovered, (less than perfectly) useful messages are written to standard error and utest exits with a non-zero return code.

 

Correctness testing within utest functions by testing copies of various sizes in the middle of an oversized buffer.  The buffers are set up as follows. Each buffer, source and destination is triple the size of the copy. Prior to the copy, a reference source and reference destination buffer which reflect the correct result are also created.  Each buffer (or section) is filled with distinct patterns of non-overlapping character sets.  The source is filled with numbers outside the section to be copied, and letters within the section to be copied.  The reference source buffer is filled with the identical contents, such that the source buffer can be tested for corruption following a memcpy.  The destination buffer is filled with the shifted number keys (in repeating pattern.  The reference destination buffer is filled with shifted number keys outside the section to received data and, letters within the section to be copied in the same pattern as the source buffer.  For each memcpy algorithm, the buffers were filled, copied and compared to the reference versions. The glibc memcpy routine was also tested as a control. The fill-copy-compare sequence is repeated over a range of sizes.

Performance testing

Each algorithm was test using ptest (ptest.c).

 

./ptest -h
Usage: ptest -s start -e end -i increment -t tries

 

The program ptest uses the timer stamp counter (rdtsc) instruction to accurately measure the amount of clock ticks a memcpy function requires. This measurement can be done over a range of copy sizes dictated by  the “-s”, “-e” (inclusive), and “-i” flag – each of which is optional. For each size, the timed memcpy is performed based on the “-t” value (optional).  Statistics are collected for each copy size and algorithm used.  For reference the i486 glibc memcpy and the i586 glibc memcpy are include in the timing tests as the “control” as above.

 

The copy of uncached source and destination data is assured by using source and destination buffers each larger than the combined caches of the Geode LX, and by advancing the start pointers for each copy through this much larger buffer in a modulo way. Additional the start pointers are calculated to start on cacheline boundaries, however this behavior maybe altered by means of a #define.

 

The statistics gathered are in reported in two distinct parts.  Detailed statistic based on a simple bucket sort of the results. These include hits per bucket, average per bucket, modes. Overall statistics report the minimum clock ticks take for a given size and algorithm and represent and attempt a simple algorithm measure of goodness.  These overall number are not uniformly repeatable and thus substantial additional post process of the detailed statistics is need.
